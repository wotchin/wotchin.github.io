大规模数据处理系统的最新技术与应用 持续更新中

2024-02-26 | 数据处理

当我们看到这个标题后，不知道大家会先想到什么。我们可能识别到的几个关键字是：架构、大规模、数据处理、最新。那么，实际上我们在这里希望探讨的话题就是最新的数据处理技术，以及这些最新的数据处理技术给实际生产带来的基于与挑战。

从数据处理说起

我首先提出一个命题——“IT 系统最核心的内容便是数据处理”，互联网尤甚。熟悉 Web 架构的读者可以知道，当前互联网架构大致可以拆分为三层：

  1. Web 前端与用户产生直接交互
  2. 中间层，用于实现业务逻辑
  3. 数据交互层，用户对数据库进行读写操作，产生数据交互

总体来说，对数据的处理可以分为两种场景：一种是 OLTP 场景，即事务处理；另一种是 OLAP 场景，即数据分析。

OLTP

OLTP 代表联机事务处理（On-Line Transaction Processing）。TP 的一个典型应用场景就是交易，例如淘宝购物、银行处理转账交易等，都属于事务型业务。数据库发展几十年来，其实最开始都是为了又快又好地实现事务（Transaction）。不要小看了 “事务”，想要实现事务的 ACID 特性其实是很复杂的，复杂到可以认为是三大基础软件（操作系统、编译系统、数据库系统）之首（普遍认为，当然也存在少量争议）。

所谓事务的 ACID 特性，代表了：

A：原子性，事务只能成功或失败，不能存在中间状态；

C：一致性，与分布式系统中的一致性含义不同，表示一个事务处理完毕后只能从一个一致的状态转移到另外一个一致的状态，也可以认为是原子性的补充；

I：隔离性，在并发事务时，要求保证每个事务必须是独立的，不能相互干扰，比较消耗性能的方法是通过锁来实现；

D：持久性，表示事务完成后，数据应该能够持久化，不能存在数据丢失。

ACID 特性保证了交易能够成功进行而不出错，如果某一个数据处理系统不能实现 ACID，那么，后果是不堪设想的。例如你选择了一家银行，它们采用的数据库不支持完全的 ACID，那么你的银行账户的余额将会神秘消失或增加。由此可见，对于传统的事务型业务来讲，对事务的支持至关重要，这是一个系统性的风险，重要程度远高于系统宕机。

OLAP

OLAP 代表联机分析处理（On-Line Analysis Processing）。如果问当前信息技术最火热的话题是什么，那毫无疑问应该是 ABC（Artificial Intelligence, Big data, Cloud computing）. 而其中的大数据（Big Data）便是数据库技术与分布式技术结合在一起的产物，或者说是适应新时代发展而产生的一种数据库形式。

虽然传统的关系型数据库听起来似乎是多数情况下用在增删改上，其实我们真正使用的业务场景中，很大程度上是读多于写的。但是不要小看 “读” 这个操作，要想实现 “读” 得更快，可比 “写” 得更快要麻烦得多。大家可能还会记得大学本科中学习过的数据库原理系列课程，知道关系型数据库是基于关系代数这样具有强数学依据的基础上建立起来的学科体系。

关系中存在投影、连接等，尤其涉及到复杂的连接（Join）和子查询等操作时，数据库的规划路径甚至可以达到 NP hard 的程度，也就是复杂到很难找到一个最优解。(这个我们在后面会讲）

而处理大数据这样的复杂业务时，往往还有一个特点，就是数据的非结构化。这是由互联网业务场景所决定的，这就导致数据虽然海量，但是价值密度却不高，如何在海量的数据中 “淘金” 是一个重要的课题。

当前比较典型的 OLAP 解决方案有：Spark、Hive、Impala、Greenplum 等。

NoSQL 的异军突起

我们在前面介绍了传统 RDBMS 的数据处理和分析方案，知道虽然传统的 RDBMS 虽然带写操作的 DML 语句具有三个（update、delete、insert），而仅具有读操作的语句只有 select 一个，而其实传统的 RDBMS 仍然具有很强的 AP 能力，这主要是来自数据库内核中优化器（又叫规划器，负责寻找最优的计划执行路径）。

伴随着互联网的兴起，人们产生的更多数据其实是非结构化的，如文本、图片、语音等，而实际上的结构化数据比重反而变少了。同时，海量的数据增长，对数据存储系统的扩展性（Scalable、Scalability）提出了要求。也就是说，传统的 RDBMS 其实是单机的，要想将单机变成分布式的，这要结合很多技术点。

在这样的一种需求催动下，NoSQL 诞生了。相比传统的 RDBMS 而言，NoSQL 具有这样的特点：

可扩展性强（High Scalability）

可用性强（High Avalibility、Zero Downtime）

云原生（Cloud Native）

上面的第三点并不能算是典型特点，只是具体的实现形式而已。而前两点确实是 NoSQL 相对于 RDBMS 的独特之处。你可能也会问，那么例如 MySQL 不是也能够具有主备部署的形式吗？这个我们等下说一下。

NoSQL 要想真正发挥实力，其实和分布式是分不开的。你也可以说，NoSQL 本身就是为了实现分布式而实现的。说到分布式就不得不提一个重要的 CAP 定理，有经验的读者可以略过。

C：强一致性，客户从所有的分布式节点上读取到的数据应该是一样的；

A：可用性，分布式系统能够在任何时候对客户提供服务，响应客户的 query；

P：分区容错性，分布式系统是一个不稳定的系统，随时都有可能会出现某个节点连接不上，即所谓的 “网络分区”，分布式系统应该具备对这样高概率事件的容错能力，即系统中部分节点网络中断时，系统仍然能够对外提供服务。

我们说一下几个典型的实例：

CA：传统的 RDBMS，由于是单机的，自然不存在分区容错

AP：MongoDB 等就是 AP 系统，要求数据强一致，适合某些简单的交易等

CP：如 Cassandra 便是舍弃强一致性，而换区高可用。Cassandra 的应用场景主要是社交网络，例如用户的信息详情页这种，即便看到的数据不是最新的，其实也无妨，总比看到 502（指代服务不可用）强得多。

传统技术不足与 NewSQL 的诞生

我们前面提到了 NoSQL 与传统的 RDBMS 两种数据库，其中 NoSQL 的可扩展性和高可用性是传统 RDBMS 的不足，而传统的 RDBMS 具有的 SQL 语句和对事务的支持，以及复杂的查询关系又是 NoSQL 的不足。

同时，当前处理大数据（PB 级别）的 OLAP 系统，如 Hadoop 只支持读，不支持写。公司在做数据处理的时候，需要定期地将业务系统中的数据同步到冷数据存储系统（指不经常查询的系统，这里特指使用 Spark 等 OLAP 工具处理数据的数据来源）或大数据存储系统（如 Hadoop 的 HDFS 或亚马逊 S3）中，实际上需要维护两套环境，一套是高并发业务访问的环境，一般是 TP 场景；另一套是重计算的 AP 分布式集群中。而且，跑 OLAP 作业的时间一般比较长，往往是晚上通过任务调度系统（如 Azkaban）来触发。这就造成其实集群的使用成本高，即白天 AP 系统空闲率高，夜晚 TP 系统空闲率高。

那么，能否具有一种通用的系统，既能够实现海量数据的离线计算或者即席查询（Hive 的即席查询能力很弱），又能支持混合型的 workload（负载），同时还可以满足高并发的 TP 场景需要，且能够支持完整的 ACID 事务，同事还需要融合 NoSQL 的高可用、可扩展性。

那么，这样吹毛求疵的系统能不能实现？原先的确不能！但是，随着技术的发展，如最近几年 Raft、Paxos 等分布式共识算法的提出与实现、跨 AZ 高可用技术的诞生、基于 GPS 和原子钟的全球同步时钟等技术的诞生，一种新型的数据处理系统诞生了，将其称之为 NewSQL。

简而言之，NewSQL 应该是这样的：

支持混合型的 workload，OLAP + OLTP = HTAP

支持 SQL 语句，具备关系模型，支持 ACID 事务

高可用、线性可扩展、强一致性

既能够支持高并发场景，又能够 cover 海量数据即席查询

NewSQL 听起来好 NB，有哪些具体的实现？如 Google 的 Spanner/F1、Amazon 的 Aurora，以及 PingCAP 的明星产品 TiDB 等。

MPP 架构

基于原生 PostgreSQL 实现的 Greenplum（硅谷著名公司 Pivotal 开源产品，Java 框架 Spring、SQL on Hadoop 的 Apache Hawq 也是出自该公司），将数据库优化器产生的执行计划拆分为不同的子作业，将这些子作业下发到不同的数据节点上进行计算，即可实现分布式并行计算，称之为 MPP 架构。该架构既可以实现即席查询，又可以实现数据的增删。

PGXL 也是基于原生 PostgreSQL 实现的 MPP 分布式数据库，感觉在实现上是山寨的 Greenplum，采用类似架构的还有腾讯云最近搞出来的 CynosDB 等。

这种 MPP（Massive Parallel Processing）架构的特点是 share nothing 的，所有的数据节点数据不共享，自己存储自己的；同时讲集群中的节点分为两大类，分别是 Coordinator Node（CN）与 Data Node（DN）。

CN 负责生成计划，DN 负责执行计划。

这就不得不介绍一下，我们前面提到过的数据库内核，我们在这里简单介绍一下这个现代 IT 系统最重要的组件之一，Web 后端的 nucleus——数据库系统的内核。

数据库系统内核概述
数据库内核可以分为两大块，分别是 SQL 引擎与存储引擎，SQL 引擎又分为解析器、查询优化器、执行器，存储引擎又分为事务、元数据存储、缓冲池、锁管理等。

上面所述的 MPP 架构是一种 shared-nothing 的体系结构，通过 coordinator 生成分布式执行计划，再通过计算引擎 Data Node 执行计算。本质上是将 SQL 引擎和存储引擎拆开，分别放置在两种不同的节点上。这种类似架构其实在上世纪就已经有类似的雏形了。

NewSQL 的技术特点

NewSQL 能够解决的问题我们前面已经介绍过了，它能够解决 NoSQL 和传统的 RDBMS 的不足，综合二者的长处。我们现在介绍一下 NewSQL 最近所使用的技术点：

  1. 普遍采用新硬件
  2. 跨 AZ 抵抗重大灾难的容灾能力
  3. 基于分布式系统的最佳实践

普遍使用新硬件

这一点主要在云原生数据库上体现更为明显。例如阿里的 PolarDB，支持 RDMA 这种新型的网络硬件设备，该设备能够直接从对方 host 的内存中读取数据，因此，数据的传递效率很高，避免了层层传递的消耗。

另一方面，如谷歌的 Spanner 采用了全球时钟，这其实是一种投机取巧的方法，采用新硬件来实现数据的同步，而不是采用分布式一致性的算法来实现。在实现上通过 GPS 的全球授时服务，在每台 Node 上获取时间，然后通过超级精度的原子钟，来同步计时。因此，理论上讲，即使过上几百万年，时间的误差也很小。由于每个节点上的时间是完全一样的，在发送信息的时候带上一个时间戳，就可以通过时间戳来同步事件了。

跨 AZ 重大灾难的容灾能力

AZ 指的是 Available Zone，是指一个可用的区域，譬如广州的机房和北京的机房就可以说是两个不同的可用区。

那么要想获得极大的容灾能力，就要求 AZ 要尽可能地分布广泛，例如在华盛顿和纽约都部署了机器，虽然可以抵抗纽约的地震，但是却不能抵抗美国的内战。但是如果节点放在中国和美国，那么荣在能力就更强了。但是，这同时又对通信提出了高要求，因为这种情况下，虽然可用性得到了提高，但是节点之间的同步成本将会提高，而且这种 NewSQL 又是强一致性的系统，那么势必可能会导致 CAP 中的可用性Ａ受到影响。

分布式系统是当前互联网技术的一个关键点，NewSQL 数据库都在不同程度上对原生 Paxos 协议进行了改进，如 TiDB 使用的是新的 Raft 协议，更简洁。同时，NewSQL 要求的是强一致性。这一点上其实没什么太多新的东西。

数据库内核的创新

我们在前面介绍过数据库的内核，其实数据库技术的工业界和学术界的差距不大，基本上是同步的状态。我们在前面介绍过数据库内核的基本结构。拥抱 SQL 是几乎近些年的所有实践的一个方向，例如 Spark SQL、Kafka 支持 SQL 接口，SQL on Hadoop 等。

那么，提到 SQL 就不得不提到 SQL 引擎。

向量化执行引擎

最新的执行器会采用一种 lazy 的形式，将多个执行操作暂存起来，进行批量化执行，这样既可以提高执行效率。

Codegen 技术

传统的执行器通常是火山模型，数据来一条执行一条，这样就会导致 CPU 的 Cache 命中率低，是一种 Cache 不友好的行为。那么是否可以动态生成对 CPU Cache 友好的代码呢？这是可以的，这就是所谓的 Codegen 技术。

上面两点技术在 Spark 的最新版本上已经实现，是两种比较崭新且高级的对数据库内核的优化方法。

自治数据库

自治数据库这个概念其实早在上世纪就已经提出来了，只不过直到最近的这几年才有真正落地的模样，而且这也是目前的一个研究热点，大规模落地的方案还不太多。

通常地，自治数据库是采用机器学习的技术用于优化数据库的管理、调度、调优、安全检测等。

微软算是最早的一波自治数据库的研究公司，目前云上 Azure 支持自调优等自治功能。Oracle 的云上数据库，据他自己说是第一个实现自治数据库的产品（不知道其他厂商会怎么想），全方位支持自安全、自管理、自调优等，号称不需要 DBA 就可以实现超高性能。

虽然不知道自治数据库的未来会怎么发展，但是目前来看，机器学习这么火热，其在数据库等场景上的应用还不足，当前主流的数据库 & 数据挖掘顶会 SIGMOD 等上面已经出现很多关于机器学习与数据库结合的文章了，而且 AI 方面的顶会上也都有相关研究结果。

这里面比较典型的论文有 Google 提出的自动建索引方法以及 SageDB 这种基于 AI 的全新架构。

总结

我们在这篇文章中，简明地回顾了近些年数据处理领域的关键核心技术。其实做到数据库内核中的创新并不是那么多了，这个领域其实已经很成熟了。未来的几个方向我觉得还是解决优化器的最佳规划执行路径的查找，这应该是一个 NP-HARD 问题；分布式系统、分布式最佳执行计划生成等技术点。

我们可以看到，NewSQL 结合了 NoSQL 和 RDBMS 的优点于一身，国内诞生了蚂蚁金服的 OceanBase 以及 TiDB 这样的优秀 NewSQL 数据库，具有出色的 TP 和 AP 能力；新硬件是一个全兴的概念，这对云原生数据库是一个机遇与挑战；结合 AI 技术的数据库或者说数据处理解决方案是一个热点，也是目前的一个风口，值得关注。
